# Mendix Container Resize Case Study

This document presents a research-oriented summary and operational guide for using a maintenance page during a Mendix container resize event.

---

# Mendix Maintenance Page Strategy
**Focus:** Infrastructure resilience and end-user continuity during Mendix container scaling.

---

## 1. Core Concepts
A successful maintenance page strategy depends on understanding two separate layers:

* **Application Runtime:** The Mendix app process running inside a container.
* **Container Infrastructure:** The host environment that can be replaced or resized independently.
* **Maintenance Page:** A network-layer response served by the cloud edge or load balancer, not by the application itself.

This separation means the maintenance page can remain available even when the application container is offline or being recreated.

---

## 2. Maintenance Page vs Application Error Pages
Many beginners confuse a maintenance page with an application error page such as `404.html` or `offline.html`.

* **Maintenance Page:** Served by the network edge or load balancer before the request reaches the application.
* **Application Error Page:** Generated by the application runtime after the request reaches the app and the app decides the response.

### Key differences
* **Where it is served:**
  * Maintenance page = cloud edge/load balancer
  * 404/offline page = application runtime or web server
* **When it is used:**
  * Maintenance page = planned downtime, container rebuild, or infrastructure maintenance
  * 404/offline page = missing page, app error, or local app failure
* **What it depends on:**
  * Maintenance page = independent assets and network routing
  * 404/offline page = application resources, theme, or runtime availability

When the whole container is under maintenance, the maintenance page is the correct mechanism because the application itself may not be available to serve a normal error page.

---

## 3. Design Principles for the Maintenance Page
For reliability during downtime, the custom page should be self-contained and avoid dependencies on the application.

### Recommended constraints
* **Single HTML file:** Keep the page compact and simple.
* **Inline CSS:** Include all styling in a `<style>` block to avoid external asset loading.
* **Standalone images:** Use embedded Base64 assets or stable CDN-hosted URLs for logos and illustrations.
* **No runtime app scripts:** Do not reference application-generated JavaScript or dynamic resources.

These principles help ensure the page renders correctly even when the application is unavailable.

### Example maintenance page structure
A simple maintenance page can look like this:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Maintenance in progress</title>
  <style>
    body { font-family: Arial, sans-serif; margin: 0; background: #f4f6f8; color: #333; }
    .page { display: flex; flex-direction: column; justify-content: center; align-items: center; min-height: 100vh; text-align: center; padding: 24px; }
    .logo { max-width: 220px; margin-bottom: 24px; }
    .card { background: #fff; border-radius: 12px; padding: 28px; box-shadow: 0 12px 36px rgba(34, 40, 49, 0.08); max-width: 600px; width: 100%; }
    h1 { margin-bottom: 12px; }
    p { margin: 0 0 16px; line-height: 1.6; }
  </style>
</head>
<body>
  <div class="page">
    <div class="card">
      <img class="logo" src="https://example.com/logo.svg" alt="Brand logo">
      <h1>We’ll be back shortly</h1>
      <p>We are updating the system to improve performance. Thanks for your patience.</p>
    </div>
  </div>
</body>
</html>
```

This example uses only HTML and inline CSS, with no application-specific assets or scripts.

## 4. Operational Workflow
The maintenance page is typically enabled at the load balancer or cloud edge and does not require restarting the Mendix runtime.

### Typical sequence
1.  **Prepare:** Upload the maintenance HTML file to the cloud portal or edge configuration.
2.  **Enable:** Turn on maintenance mode shortly before the resize window.
3.  **Resize:** The infrastructure team replaces the container while the maintenance page remains available.
4.  **Verify:** Monitor logs and runtime health indicators.
5.  **Disable:** Turn off maintenance mode once the application is healthy.

This workflow helps preserve user-facing branding while the container is scaled.

---

## 5. Validation Approach
Validating the custom maintenance page in a non-production environment helps catch implementation issues early.

### Recommended validation steps
1.  **Visual check:** Enable maintenance mode and confirm layout, typography, and imagery on desktop and mobile.
2.  **Simulated downtime:** With maintenance mode active, stop the environment or container and verify the page still displays correctly.
3.  **Cache check:** Refresh in a private browser session to ensure the edge serves the latest HTML file.

These checks help ensure the page works as a true fallback.

---

## 6. Simplified Stakeholder Summary
| Action | Impact on Runtime | Requires Restart? | User Sees |
| :--- | :--- | :--- | :--- |
| Enable maintenance mode | No change to runtime | No | Custom maintenance page |
| Resize container | Application container recreated | Yes (infrastructure) | Custom maintenance page |
| Disable maintenance mode | Runtime resumes normal traffic | No | Live application |

---

## 7. Smoke Testing After Resize
After the resize completes, confirm that the Mendix runtime has started and verify application behavior before returning traffic.

### Smoke test options while maintenance is enabled
* **Alternative environment URL:** Use any technical or default Mendix environment URL that may bypass the primary maintenance setting.
* **Controlled bypass:** If supported, use a temporary request header or internal access mechanism to perform validation.
* **Health endpoint:** Query a health-check endpoint if available.

If bypass access is not available, briefly disable maintenance mode during a low-traffic window to confirm the application responds and then re-enable it if needed.

---

## 8. Load Balancer and Edge Validation
Confirm the maintenance page is served by the cloud edge rather than the application runtime.

### Useful checks
1.  Resolve the application URL and verify it points to the load balancer or cloud edge.
2.  Inspect the HTTP response headers for edge-related server identifiers.
3.  Confirm the response content is consistent with the maintenance HTML file.

These checks validate the architecture and confirm the page is functioning as intended.

---

## 9. Recommended QA Workflow
1.  **Resize completion:** Confirm the infrastructure change has finished.
2.  **Runtime verification:** Check logs or health indicators for Mendix startup.
3.  **Access validation:** Use a supported bypass path or endpoint to verify the app is reachable.
4.  **Functional check:** Confirm core application operations and storage access.
5.  **Restore traffic:** Disable maintenance mode when the application is ready.

---

## 10. Common Pitfalls
* Assuming `404.html` or `offline.html` will work when the entire container is unavailable.
* Embedding application-specific assets or scripts that fail when the runtime is down.
* Relying on cached browser content instead of confirming the edge serves the latest maintenance HTML.

## Final Notes
A maintenance page strategy provides a stable user-facing experience while container infrastructure changes occur. The key is to keep the page independent of application resources and to validate it in a lower-risk environment before production use.
